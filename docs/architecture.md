# Copilot-First Toolkit Architecture

## Principle

Keep the public surface narrow, the internal operating system deep, and repository memory file-based and compact.

- Two public agents by capability tier: `team-lead` and `developer`
- Hidden specialists for architecture, implementation, review, DB, migration, and bootstrap
- Repo-context skills for durable domain rules and guardrails
- Repo-local `.github/memory/` for compact living repository context
- `.github/` as the only active workspace runtime root
- `.ai/memory/` as generated workspace inventory only
- Context compression before prompt expansion: selective loading, compact summaries, isolated sub-task outputs

---

## Canonical Layout

```text
.github/agents/          ← runtime agents
.github/skills/          ← runtime skills
.github/prompts/         ← runtime prompts
.ai/memory/              ← generated workspace inventory and MCP registry
<repo>/.github/memory/   ← compact repository memory
.ai-devtoolkit/          ← reusable source catalog
```

The toolkit is copied into `.github/` by initialization and then maintained through bootstrap and catalog audit scripts. Repository memory lives inside each repository so it stays close to the code it describes without duplicating the workspace runtime.

---

## Runtime Contract

### Public surface

The public Copilot surface is intentionally small:

- `team-lead` for premium orchestration with hidden specialist delegation
- `developer` for bounded direct execution on smaller paid models

Responsibilities:
- classify the user request
- inspect the workspace before committing to a plan
- choose the right workflow or hidden specialist
- load only the repo context that is relevant to the task
- run review and fix loops until the result is coherent
- summarize outcome, validation, and remaining risks

### Internal surface

All other agents are hidden specialists. They can be added to `team-lead` delegation lists, but they are not meant to be invoked directly by default.

### Repo knowledge model

Repository-specific context is split across two layers:

```text
.github/skills/<workspace>-<repo>/
  SKILL.md
  references/guardrails.md
  assets/domain-rule.template.md

<repo>/.github/memory/
  context.md
  dependencies.md
  recent-changes.md
```

- Repo-context skill: durable domain vocabulary, business rules, and guardrails.
- `context.md`: stable facts, entry points, known traps, and migration notes kept intentionally short.
- `dependencies.md`: compact technical map of modules, external systems, REST surfaces, DB signals, Kafka signals, and legacy UI entrypoints.
- `recent-changes.md`: compact recent commit and working-tree summary for fast rehydration.

This preserves multi-repo bootstrap without multiplying the visible agent catalog and brings the runtime closer to Claude Code-style file memory without introducing a separate semantic memory store.

## Memory Layers

| Layer | Scope | Owner | Contents | Churn |
|------|-------|-------|----------|-------|
| `.ai/memory/*.json` | workspace | bootstrap engine | repo inventory, MCP registry, adapter state | regenerated |
| `.github/skills/<workspace>-<repo>/` | workspace runtime | developer | durable repo rules and vocabulary | low |
| `<repo>/.github/memory/context.md` | repo | developer | compact stable facts and traps | medium |
| `<repo>/.github/memory/dependencies.md` | repo | generated by repo-memory refresh | technical dependency map | medium |
| `<repo>/.github/memory/recent-changes.md` | repo | generated by repo-memory refresh | compact recent deltas | high |

Rules:
- Do not store business rules in `.ai/memory/`.
- Do not duplicate generic Quarkus, Java, Kafka, or testing rules into repo memory when a shared skill already owns them.
- Do not duplicate workspace runtime assets inside repo-local `.github/`; repo-local `.github/memory/` is the allowed repo memory surface unless the repo already owns additional assets.
- Keep repo-memory files short enough to read before a task without bloating the context window.

---

## Context Loading Order

Use this order when the task is scoped to a repository:

1. Read `.ai/memory/workspace-map.json` only to identify repositories and local agent assets.
2. Read `.github/skills/<workspace>-<repo>/SKILL.md` for durable repo rules.
3. Read `<repo>/.github/memory/context.md` for compact stable facts.
4. Read `<repo>/.github/memory/dependencies.md` for technical surface and integration signals.
5. Read `<repo>/.github/memory/recent-changes.md` only when recent implementation deltas matter.

Compression rules:
- Prefer repo-memory summaries over re-reading large docs or logs.
- Keep change journals newest-first and compact.
- Return compact results from delegated work instead of forwarding raw outputs.
- Refresh repo memory after structural changes or before large investigations when the files are stale.

## Initialization Flow

### Phase 0 — New Project Initialization

Command:

```bash
node .ai-devtoolkit/scripts/new-project.mjs --name ... --domain ... --repos ...
```

Effects:
1. Copy generic agents into `.github/agents/`
2. Copy generic self-contained skills into `.github/skills/`
3. Keep `team-lead` and `developer` as the only public agents
4. Generate repo-context skills for each repository
5. Scaffold `<repo>/.github/memory/` for repositories that already exist
6. Create `AGENTS.md` and `package.json`

### Phase 1 — Workspace Bootstrap

Command:

```bash
npm run bootstrap:ai
```

Effects:
1. Detect runtime mode
2. Ensure `.github/agents`, `.github/skills`, and `.github/prompts` exist
3. Prune previously managed legacy adapter paths when present
4. Refresh `.ai/memory/workspace-map.json`
5. Refresh `.ai/memory/mcp-registry.json`

### Phase 2 — Project Readiness Audit

Command:

```bash
npm run bootstrap:project
```

Checks:
1. Agent catalog completeness
2. `team-lead` and `developer` are the only public agents
3. Each repository has a repo-context skill
4. Each repository has repo-memory files in `<repo>/.github/memory/`
5. Required MCP servers are configured

### Phase 3 — Repo Memory Refresh

Command:

```bash
npm run memory:refresh
```

Effects:
1. Ensure repo-memory directories exist for detected repositories
2. Refresh `dependencies.md` from local build and source signals
3. Refresh `recent-changes.md` from git history and working-tree status
4. Leave `context.md` developer-owned and compact

## Environment Inventory

The bootstrap inventory writes this environment shape:

```json
{
  "mode": "IDE | TERMINAL",
  "primary_tool": "copilot",
  "secondary_tools": []
}
```

The runtime is intentionally Copilot-only. Older adapter roots may still exist in historical workspaces, but the active architecture no longer depends on them.

## File Ownership

| File or folder | Owner | Behavior |
|----------------|-------|----------|
| `.github/agents/*.agent.md` | workspace runtime | copied from toolkit, then developer-owned |
| `.github/skills/<name>/` | workspace runtime | copied from toolkit, then developer-owned |
| `.github/skills/<workspace>-<repo>/` | developer | generated once, then repo-context owned |
| `<repo>/.github/memory/context.md` | developer | created once, then manually curated |
| `<repo>/.github/memory/dependencies.md` | repo-memory refresh | regenerated when refreshed |
| `<repo>/.github/memory/recent-changes.md` | repo-memory refresh | regenerated when refreshed |
| `.ai/memory/*.json` | bootstrap engine | regenerated |
| `AGENTS.md` | developer | created once, then developer-owned |
| `.vscode/mcp.json` | developer | manually configured |

## Why This Shape

This architecture keeps the Copilot user experience simple while preserving a strong internal operating system:

- less catalog noise for the user
- reusable specialist knowledge retained internally
- deterministic workflows for non-trivial work
- compact file-based repository memory for fast context rehydration
- generic bootstrap preserved for new multi-repo workspaces
- no project-specific public routing model baked into the toolkit
